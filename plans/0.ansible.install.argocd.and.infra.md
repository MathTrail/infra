План: "The Core Orchestrator" (Hybrid Ansible Framework)
1. Мотивация: Какую проблему мы решаем?
Проблема: До этого момента у нас была «каша» из Bash-скриптов, раскиданных по разным репозиториям. Чтобы развернуть систему, нужно было помнить последовательность, заходить в разные папки и надеяться, что локальные версии файлов в ../infra совпадают с тем, что ожидает skaffold.

Хрупкость: Ошибка в одной строке Bash ломает всю цепочку, и трудно понять, где именно.

Связность: Core слишком много знал о деталях (например, как ждать Vault), хотя это забота репозитория Infra.

Отсутствие CI-готовности: Локальный деплой слишком сильно отличался от того, как это должно работать на сервере.

Решение: Ansible в Core становится «единой точкой правды». Он не знает как работает база в Infra-Local, он просто дает команду: «Развернись». При этом он умеет работать и с локальными папками (для скорости), и с GitHub (для надежности).

2. Архитектура "Черного ящика" (Encapsulation)
Каждый репозиторий инфраструктуры (Infra, Infra-Local, Infra-Local-K3s) теперь обязан иметь:

ansible/deploy.yml — входной файл для деплоя этого конкретного модуля.

ansible/vars/main.yml — дефолтные переменные этого модуля.

Core вызывает их, прокидывая общие переменные (Registry URL, Domain, Env) через --extra-vars.

3. Компоненты реализации (Step-by-Step)
Этап А: Подготовка Core (Оркестратор)
Core/ansible/vars/cluster_config.yml: Создаем реестр "Желаемого состояния". Здесь прописаны ветки (branches) для каждого репозитория.

Core/ansible/roles/runner: Создаем "умную" роль.

Если deploy_mode == local: проверяет наличие папки ../../repo-name.

Если deploy_mode == remote: делает git clone ветки из GitHub во временную директорию.

Запускает ansible-playbook внутри целевой папки.

Этап Б: Автономизация дочерних репозиториев
Infra-Local-K3s: Переносим логику создания кластера в ansible/deploy.yml. Он должен уметь проверять, создан ли кластер, чтобы не падать при повторном запуске.

Infra-Local: Переносим установку Postgres/Redis/Kafka в ansible/deploy.yml (используем модуль kubernetes.core.helm).

Infra: Переносим установку ArgoCD и тот самый цикл ожидания Sync Waves (0-4) в ansible/deploy.yml.

Этап В: Унификация через DevContainer
Добавляем в .devcontainer/devcontainer.json установку ansible и kubernetes.core коллекции. Теперь у всех разработчиков "пульт управления" работает одинаково.

4. Логика работы Hybrid Runner (Pseudocode)
YAML
# Внутри роли runner в Core
- name: "Set path"
  set_fact:
    target_path: "{{ (mode == 'local') | ternary('../../' + repo, '/tmp/' + repo) }}"

- name: "Git Sync (if remote)"
  git: repo=... dest={{ target_path }} version={{ branch }}
  when: mode == 'remote'

- name: "Run Sub-Orchestrator"
  command: "ansible-playbook ansible/deploy.yml"
  args:
    chdir: "{{ target_path }}"
5. Итоговый Workflow для разработчика
Локальная правка: Ты изменил конфиг Kafka в ../infra-local.

Запуск: В корне Core пишешь just deploy.

Результат: Ansible в Core видит, что ты в local моде, идет в соседнюю папку ../infra-local, запускает там ansible/deploy.yml, и твоя Kafka обновляется за 10 секунд.

Проверка перед релизом: Пишешь just deploy-remote --branch develop. Ansible выкачивает develop ветки из GitHub и проверяет, что всё собирается "на чистую".

Почему это "Киллер-фича"?
Изоляция: Ты можешь полностью переписать Infra, не трогая Core.

Скорость: Тебе не нужно пушить в Git, чтобы протестировать инфраструктурный код.

Масштабируемость: Нужно добавить систему мониторинга? Создаешь репозиторий Infra-Monitoring, добавляешь одну строчку в Core/ansible/main.yml, и готово.
